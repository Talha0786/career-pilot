# 017 — Known limitation: simultaneous embed redelivery can double-call the LLM

**Status:** DONE | **Milestone:** M2

## Objective
`embedJobPosting` short-circuits on redelivery ONLY when the first delivery
has already committed `embeddingStatus='ready'` before the second delivery's
read. Under genuinely SIMULTANEOUS redelivery (both read 'pending' before
either writes), both calls reach the LLM and both get billed — final DB
state is still correct (JobPosting.attachEmbedding is idempotent per-model,
task 003), but the cost-dedup guarantee is only proven for sequential
redelivery, not a true race. Same class of gap as tasks 015/016.

## Files affected
- `packages/application/src/discovery/commands/embed-job-posting.ts`
- `packages/infrastructure/src/llm/postgres-budget-store.ts` (reuse the
  advisory-lock pattern, keyed on jobPostingId instead of userId)

## Dependencies
016 (establishes the advisory-lock-around-use-case pattern)

## Acceptance criteria
- [x] `embedJobPosting` takes an optional lock around read-check-embed-write,
      keyed on `jobPostingId`
- [x] A genuine concurrency test (two BullMQ deliveries enqueued and started
      as close to simultaneously as the test can force) results in exactly
      ONE call to the LLM, not "eventually consistent state with possibly two"

## What was built
- `JobPostingRepository.withJobPostingLock?<T>(jobPostingId, fn)` — new optional port method ([repositories.ts](../packages/application/src/ports/repositories.ts)), same shape as `BudgetStore.withUserBudgetLock` from task 016.
- `DrizzleJobPostingRepository.withJobPostingLock` — implemented with `pg_advisory_xact_lock(hashtext(jobPostingId))` inside a `db.transaction`, mirroring `PostgresBudgetStore.withUserBudgetLock` exactly.
- `embedJobPosting` — the read-check-embed-write body was extracted into a `doEmbed` helper; the exported use case now wraps it in `deps.jobPostings.withJobPostingLock(...)` when the repository supports it, falling back to unlocked execution otherwise (so fakes/mocks without the method are unaffected).

## Test plan actually executed
Replaced the `it.todo(...)` placeholder in `apps/worker/test/integration/end-to-end.test.ts` with a real test: two BullMQ deliveries for the same `jobPostingId` enqueued back-to-back with **no wait** between them (unlike task 010's sequential-redelivery test), against a counting fake LLM with an artificial 200ms delay to widen the race window. Asserts `callCount === 1` and exactly one `ai_invocations` row.
- **Regression-proof**: temporarily renamed `withJobPostingLock` on the Drizzle repository so the use case fell back to unlocked execution, reran the test in isolation — it failed with `expected 2 to be 1` (the real, previously-documented bug). Restored the method, reran — passes. This confirms the test actually exercises the race rather than passing vacuously.
- Full suite after the fix: 162 unit + 38 integration (was 37 + 1 todo) all pass. `pnpm -r typecheck` clean across all 8 packages.

## Note
Closes the gap that was deliberately left open in M2's original scope, once the pattern from tasks 015/016 (advisory-lock-around-use-case) existed to reuse. Same lock-ordering discipline as budget-guard: the jobPosting lock is always the outermost lock (embed-job-posting wraps, budget-guard's user lock is nested inside via `deps.llm.embed`), so no lock-ordering deadlock is possible across the two independent advisory-lock keyspaces.
