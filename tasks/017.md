# 017 — Known limitation: simultaneous embed redelivery can double-call the LLM

**Status:** TODO (documented, not fixed) | **Milestone:** M2 (tracked for M5+)

## Objective
`embedJobPosting` short-circuits on redelivery ONLY when the first delivery
has already committed `embeddingStatus='ready'` before the second delivery's
read. Under genuinely SIMULTANEOUS redelivery (both read 'pending' before
either writes), both calls reach the LLM and both get billed — final DB
state is still correct (JobPosting.attachEmbedding is idempotent per-model,
task 003), but the cost-dedup guarantee is only proven for sequential
redelivery, not a true race. Same class of gap as tasks 015/016.

## Files affected
- `packages/application/src/discovery/commands/embed-job-posting.ts`
- `packages/infrastructure/src/llm/postgres-budget-store.ts` (reuse the
  advisory-lock pattern, keyed on jobPostingId instead of userId)

## Dependencies
016 (establishes the advisory-lock-around-use-case pattern)

## Acceptance criteria
- [ ] `embedJobPosting` takes an optional lock around read-check-embed-write,
      keyed on `jobPostingId`
- [ ] A genuine concurrency test (two BullMQ deliveries enqueued and started
      as close to simultaneously as the test can force) results in exactly
      ONE call to the LLM, not "eventually consistent state with possibly two"

## Test plan
Integration: fire two deliveries with no artificial delay between them
(unlike task 010's sequential test); assert callCount === 1, not just final
DB state correctness. This is the harder version of the test already
written and passing for the sequential case.

## Note
Deliberately left open rather than silently fixed or silently ignored. The
cost impact is bounded (at most 2x on the rare true race, not unbounded) and
M2's acceptance criteria don't require closing it — but a codebase claiming
"idempotent" should have this gap on the record, not discovered by a user's
LLM bill.
