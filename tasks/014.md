## What was built
- **`.github/workflows/ci.yml`** — 9 jobs. `lint` → `typecheck` → `boundary-check` → `unit` → {`integration`, `chaos`} → `build` → `e2e-compose` chain via `needs:` (fails fast, cheapest gates first); `security` runs independently in parallel.
  - `lint`: `pnpm lint` (root `eslint .` — see bug fixed below).
  - `typecheck`: `pnpm typecheck` (turbo across all 10 workspace packages, including the new `e2e` package).
  - `boundary-check`: `pnpm verify:boundaries` (new script, see below).
  - `unit`: `pnpm test:cov` — the coverage gate IS this step; vitest exits non-zero below threshold.
  - `integration`: real `pgvector/pgvector:pg16` + `redis:7-alpine` GitHub Actions service containers (not Testcontainers-the-library — service containers give the same "real engine, no mocks" guarantee ADR-002 requires, without Docker-in-Docker); runs `pnpm db:migrate` then `pnpm test:int`.
  - `chaos`: same service containers, runs the new `pnpm test:e2e` (see chaos test below).
  - `build`: `pnpm --filter @careerpilot/web build` (the one package with a real build step) + `docker compose build` (validates all four Dockerfiles — this is what actually catches Dockerfile bugs, see task 018's postscript).
  - `e2e-compose`: brings up the real `docker compose` stack and runs a new smoke-test script against it (see below).
  - `security`: CodeQL (javascript-typescript), `osv-scanner`, `gitleaks`, `trivy` filesystem scan.
- **`.github/workflows/nightly.yml`** — cron (03:00 UTC daily) + `workflow_dispatch`; installs real Ollama, pulls `nomic-embed-text`, runs `pnpm test:nightly` against it.
- **`scripts/verify-boundary-enforcement.mjs`** — the boundary-violation test. Writes two fixture files into `packages/domain/src/` (always cleaned up, including on SIGINT/SIGTERM) and proves BOTH enforcement layers actually catch a violation: a bare-specifier `@careerpilot/infrastructure` import must fail `tsc` (TS2307, pnpm's isolated `node_modules`), and a *relative-path* import reaching across the package boundary — which `tsc` alone does NOT catch, since relative imports bypass `node_modules` resolution entirely — must fail `eslint` (`import/no-relative-packages`). Exposed as `pnpm verify:boundaries`.
- **`e2e/chaos/worker-kill.spec.ts`** + **`vitest.e2e.config.ts`** — the chaos test. Seeds 40 jobs, spawns the REAL worker binary (`apps/worker/src/main.ts` via tsx, as a genuine child process — not an in-process function call) with a small `OUTBOX_BATCH_SIZE` so the relay needs multiple poll cycles, lets it run ~900ms, sends a real `SIGKILL` (no graceful shutdown), waits for the OS to reap it, spawns a fresh worker to simulate an orchestrator restart, and asserts: zero jobs stuck `pending` (no lost events), `ai_invocations` count exactly 40 (no duplicate billing/processing), zero outbox rows left unpublished. New `e2e` workspace package (added to `pnpm-workspace.yaml`) so the spec file can resolve `@careerpilot/*` bare specifiers.
- **`scripts/e2e-compose-smoke.mjs`** — drives the real HTTP surface of the compose stack: register → login → verify session → paste a job → add it to the pipeline → poll the board until the embedding status is terminal → logout. Exposed as `pnpm test:e2e:compose`.
- **`packages/infrastructure/test/nightly/ollama-contract.test.ts`** + **`vitest.nightly.config.ts`** — real-Ollama contract test. Skips gracefully (not a failure) when no Ollama is reachable, so it's safe to leave in the tree without a dedicated CI runner catching it accidentally.

## Bugs found and fixed along the way
- **`pnpm lint` had never actually been run against this codebase** (task 001 claimed boundary lint was DONE with unchecked boxes — a stale-docs discrepancy noted at the start of this session). Running it surfaced 2 real ESLint errors (`no-empty-object-type` in `Checkbox.tsx`/`Switch.tsx`, fixed by converting empty-extends interfaces to type aliases) and ~15 unused imports across `application`/`infrastructure` (fixed by deletion). Also discovered the root `"lint": "turbo run lint"` script silently skipped `apps/api`, `apps/web`, and `apps/worker` (none define a `lint` script) — changed to `"lint": "eslint ."` so the root script actually covers the whole repo, matching what CI needs to enforce.
- **The coverage gate (`pnpm test:cov`) had also never been run before** — it failed at 80.55% against the configured 90% threshold on first execution. Root cause was two-fold, both genuine: `discovery/queries/get-job.ts` and `list-jobs.ts` had zero test coverage (fixed by adding `packages/application/test/unit/discovery-queries.test.ts`, 6 new tests), and `ports/llm.port.ts`/`ports/repositories.ts` are pure `interface`/`type` files with no runtime code to cover at all (correctly excluded from the coverage `include` set in `vitest.config.ts`, same rationale already applied to `events.ts`). Final coverage: 99.61% statements/lines, 96.75% branches, 100% functions.
- **ESLint had no Node-globals awareness for plain `.mjs` scripts** — the two new scripts (`verify-boundary-enforcement.mjs`, `e2e-compose-smoke.mjs`) tripped `no-undef` on `process`/`console`/`fetch` (TypeScript files are exempt because `typescript-eslint`'s recommended config disables `no-undef` in favor of the compiler's own checks; plain `.js`/`.mjs` files get no such exemption). Fixed by adding the `globals` package and a `languageOptions.globals: globals.node` override for `**/*.mjs`.
- **`scripts/e2e-compose-smoke.mjs`'s first draft missed the separate `POST /api/applications` call** — `POST /api/jobs` only creates the `JobPosting`; it does not put it on the board (the existing `board/page.tsx` UI always calls both `createJob` then `createApplication` — the smoke script initially called only the first, so the board's "discovered" column stayed empty and the poll for a terminal embedding status timed out). Fixed by adding the missing call; verified passing against the live compose stack afterward.

## Test plan actually executed
Per the original test plan ("deliberately break each gate once and confirm it goes red"):
- **Boundary check**: ran with the fixtures present — confirmed both fail as expected (see script). This IS the test; there's nothing else to break.
- **Chaos test**: the test's own built-in "sanity" assertion (`stillPending > 0` immediately after the kill, before the restart) proves on every run that the kill actually landed mid-backlog rather than after the worker had already finished — a test that could pass vacuously (kill happens too late to matter) would defeat the point. Ran it 4x consecutively locally against the real Docker Postgres/Redis with zero flakes.
- **Coverage gate**: ran `pnpm test:cov` before the fix (red, 80.55%) and after (green, 99.61%) — a genuine before/after, not a simulated one.
- **`e2e-compose` smoke script**: ran it against the actually-running compose stack (`docker compose up -d --build`, same stack task 013/018 were verified against) — first run failed for a real reason (missing `createApplication` call, see Bugs above), second run passed.
- **`docker compose build`**: ran locally — all four images (api, worker, web, migrate) build clean.
- Full local suite after all fixes: `pnpm lint` (0 errors/warnings), `pnpm -r typecheck` (10/10 packages clean), `pnpm test:cov` (168 tests, coverage gate green), `pnpm test:int` (38 tests), `pnpm test:e2e` (chaos test, 4 consecutive clean runs), `pnpm test:e2e:compose` (smoke test against live stack).
- **Not independently verified**: the `.github/workflows/*.yml` files themselves have not been observed running on an actual GitHub Actions runner as of this write-up (this environment has no `gh` CLI). Every step in `ci.yml` mirrors a command that WAS run and verified locally in this exact form (same scripts, same env var names, same service versions as the local Docker Compose stack), so the risk is scoped to GH-Actions-specific plumbing (service container networking, action versions, secrets) rather than the underlying logic — but that plumbing is real risk until a PR actually exercises it once.

## Known gaps / deviations
- **Branch protection was NOT configured.** Enabling required status checks and requiring 1 review on `main` is an access-control change to a shared GitHub resource — outside what an agent should do unilaterally per this project's safety rules, regardless of how routine it is. To finish this criterion, in the GitHub repo settings: **Settings → Branches → Add branch protection rule** for `main`, enable "Require status checks to pass before merging" (select the `lint`, `typecheck`, `boundary-check`, `unit`, `integration`, `chaos`, `build`, `e2e-compose` jobs from `ci.yml` once they've run at least once and are selectable), and enable "Require a pull request before merging" with at least 1 approving review.
- **"Testcontainers"** in the original acceptance criteria became GitHub Actions **service containers** instead (`postgres`/`redis` blocks in `ci.yml`) — same "real engine, not mocks" guarantee, without needing Docker-in-Docker on the runner. This mirrors the same substitution already made and documented for the local sandbox in task 007.
- The `security` job's four scanners (CodeQL, osv-scanner, gitleaks, trivy) are configured with sensible defaults but have not yet produced a real report to review (no PR has triggered them yet) — expect the first run to need minor config tuning (e.g. gitleaks allowlist entries for test fixtures, trivy severity threshold adjustments) rather than being final as shipped.
