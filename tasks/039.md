# 039 — Resume/cover-letter tailoring pipeline

**Status:** TODO | **Milestone:** M5

## Objective
Implement the highest-stakes pipeline in the system (`docs/06-agent-design.md` §3-4): given a profile's fact list (037) + a target job description, produce a tailored `StructuredDoc` (reusing M3's `ResumeDocumentContent`/`CoverLetterDocumentContent` — do not invent a new structured-doc model) constrained to *rephrase, reorder, emphasize, quantify only from provided facts* — never introduce new entities/employers/dates/titles/credentials/metrics.

## Files affected
- `prompts/tailor-resume/v1.md`, `prompts/tailor-cover-letter/v1.md` (fill in real content — stubbed in task 034) — system prompt must explicitly enumerate the fact-list constraint from docs/06 §4 point 1, and require the LLM to cite which fact ID(s) support each output bullet/sentence (this citation is what claim verification in 040 checks against — design the output schema now so 040 doesn't need a breaking schema change).
- `packages/contracts/src/tailoring.ts` (new) — `TailoringRequestSchema { profileId, jobPostingId, kind: 'resume'|'cover_letter' }`, `TailoringResultSchema` wrapping `ResumeDocumentContentSchema`/`CoverLetterDocumentContentSchema` (from `packages/contracts/src/documents.ts`) plus a per-bullet `supportingFactIds: string[]` field threaded through the structured content.
- `packages/domain/src/documents/document-content.ts` — extend entry/bullet shapes with an optional `supportingFactIds` field if not already extensible (check before assuming a breaking change is needed — prefer additive).
- `packages/application/src/tailoring/commands/tailor-document.ts` (new) — orchestrates: `compileFactList` (037) → load prompt (034) → guarded `llm.complete()` → zod-validate against `TailoringResultSchema` (one repair attempt, ADR-006 rule) → `Document.addVersion({source: 'generated', content, profileFactsHash, generationJobId})`.
- `apps/worker/src/main.ts` — worker handler for tailoring jobs (async, since this is a slow/expensive `large` model-tier call per docs/06 §3).
- `apps/api/src/routes/documents.ts` (extend) or new `apps/api/src/routes/tailoring.ts` — `POST /documents/tailor` (or match existing route-naming convention in that file) enqueues the job; a status-poll or WS update (reuse the M2 WS pattern already used for board live-updates) notifies the client when done.
- Tests: unit (fake LLM, assert facts-only constraint is *structurally* enforced — i.e. every bullet's `supportingFactIds` resolves to a real fact in the input fact list, reject/flag any that don't), integration (real Postgres, `Document.addVersion` persistence), worker integration test.

## Dependencies
- 033 (CostEstimator — `large` tier calls are the most expensive in the system, budget accuracy matters most here).
- 034 (PromptStore).
- 037 (fact list — the tailoring prompt's fact base).

## Acceptance criteria
- Output `ResumeDocumentContent`/`CoverLetterDocumentContent` validates against the existing M3 schema (reuse, not fork).
- Every generated bullet carries `supportingFactIds` referencing real fact IDs from the input fact list — this is checked structurally in code (not just trusted from the LLM), and any bullet with an empty or invalid `supportingFactIds` is treated as unsupported (feeds directly into 040's claim verification, but do the structural check here too as a first cheap gate before the expensive adversarial LLM pass).
- Malformed LLM output → one repair attempt → typed failure with raw output stored for triage (ADR-006, same rule as 038).
- `DocumentVersion.profileFactsHash` is populated from the exact fact list used (037's determinism guarantee makes this meaningful).

## Test plan
- Unit: orchestration with fake LLM returning valid + invalid (hallucinated fact ID, missing supportingFactIds) fixtures — assert the structural gate catches the invalid ones.
- Integration: real Postgres, full command → `DocumentVersion` persisted with correct `source: 'generated'` and `profileFactsHash`.
