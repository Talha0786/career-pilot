# 026 — Connector SDK: ConnectorPort, compliance classes, contract test-kit, registry

**Status:** DONE | **Milestone:** M4

## Objective
The shared contract every connector (Class A/B/C) must implement, plus the test-kit that proves a connector honors it. This is the foundation task — nothing else in M4 can start until this lands (ADR-004).

## Files affected
- `packages/application/src/ports/connector.port.ts` (`ConnectorPort`: `fetchJobs(config): AsyncIterable<RawJob>`, `complianceClass: 'A'|'B'|'C'`, `key: string`)
- `packages/connectors/sdk/contract-test-kit.ts` (shared `describeConnectorContract(factory)` — asserts: normalizes to canonical `RawJob` shape, respects rate limits/backoff, surfaces auth/config errors as typed results not throws, idempotent re-fetch)
- `packages/connectors/sdk/registry.ts` (in-memory registry connectors register into at startup)
- `packages/connectors/sdk/README.md` (how to write a connector + the 4 compliance classes, per folder-structure doc §"how to write a connector")
- `packages/connectors/sdk/package.json`, `tsconfig.json` (new workspace package — add to `pnpm-workspace.yaml`)

## Dependencies
005, 001

## Acceptance criteria
- [ ] `ConnectorPort` lives in `application/ports` (boundary rule: connectors are infrastructure-side adapters to an application-defined port, same shape as `LlmPort`)
- [ ] Contract test-kit is itself unit-tested against a trivial fake connector (proves the kit catches a deliberately broken connector — same "break it once" verification pattern as task 014's boundary check)
- [ ] Registry rejects two connectors registering the same `key`
- [ ] New `packages/connectors` workspace package passes boundary-check (no relative cross-package imports)

## Test plan
Unit: contract test-kit run against (a) a correct fake connector — passes, (b) a deliberately non-compliant fake connector — fails with a clear assertion. This is the regression proof, matching task 017's disable/re-enable pattern.

## Test plan actually executed

Ran inside an isolated `node:22-slim` Docker container (no local node/pnpm on this machine), full worktree tarred in, `pnpm install --no-frozen-lockfile`, then:

- `tsc --noEmit -p packages/application/tsconfig.json` → exit 0
- `tsc --noEmit -p packages/connectors/tsconfig.json` → exit 0
- `eslint packages/connectors/src packages/application/src` → exit 0 (4 warnings, `no-explicit-any` in the registry's internal `Map<string, ConnectorPort<any>>` — deliberate, a registry storing heterogeneous connector configs needs an existential type; 0 errors)
- `vitest run --config vitest.config.ts` (full unit suite, not just the new files) → **175/175 tests passed**, including:
  - `packages/connectors/src/sdk/registry.test.ts` (3 tests): register/retrieve, duplicate-key rejection, unregistered-key lookup
  - `packages/connectors/src/sdk/contract-test-kit.test.ts` (4 tests) — the "break it once" proof: a correct fake connector passes with zero failures; a connector that throws synchronously instead of returning a typed error is caught (`failures` includes `"threw synchronously"`); a connector yielding a non-canonical `RawJob` (empty `externalId`/`title`) is caught; a connector whose `configSchema` accepts a config the fixture declares invalid is caught.
- `node scripts/verify-boundary-enforcement.mjs` → both boundary layers (tsc TS2307, eslint `import/no-relative-packages`) still correctly catch violations after adding the new package.

Design notes / deviations from the literal file list in "Files affected": `ConnectorPort` lives at `packages/application/src/ports/connector.port.ts` as specified. The connector packages live under a single workspace package `packages/connectors` (matching `docs/03-folder-structure.md`'s `connectors/src/{sdk,class-a,class-b,class-c}` layout and task 026's own acceptance criterion "New `packages/connectors` workspace package") rather than as a separate `packages/connectors/sdk` package — `sdk/`, `class-a/`, etc. are subfolders of that one package, so `packages/connectors/sdk/contract-test-kit.ts` became `packages/connectors/src/sdk/contract-test-kit.ts`. `README.md` lives at `packages/connectors/README.md` (covers the SDK + all four compliance classes + "how to write a connector"), per the folder-structure doc's placement.

`ConnectorPort.configSchema` uses `zod` (`ZodType<TConfig>`) — added `zod` as a dependency of `@careerpilot/application`, mirroring `@careerpilot/contracts`' existing use of zod for DTOs; it's a pure validation library, not an infrastructure concern, so this doesn't violate the domain/application/infrastructure boundary.
