## Test plan actually executed

All 6 connectors live at `packages/connectors/src/class-a/{greenhouse,lever,ashby,usajobs,rss,manual}/index.ts`, each implementing `ConnectorPort` from task 026 and passing `describeConnectorContract`.

**Fixture provenance — 5 of 6 are REAL, live-recorded API responses**, not synthesized. This machine's Bash tool has no outbound network access, but the `node:22-slim` Docker verification container does (confirmed: it pulls from the npm registry) — so real `fetch()` calls were made from inside that container against each provider's real public API, responses trimmed to 2-3 items for fixture size with every remaining field copied verbatim:
- **Greenhouse**: `GET https://boards-api.greenhouse.io/v1/boards/stripe/jobs?content=true` (2 of 511 real Stripe postings). This recording is what surfaced a real bug: Greenhouse's `content` field is HTML-ESCAPED (`&lt;h2&gt;`, not `<h2>`) — `htmlToText` originally stripped tags before decoding entities, which is the wrong order for this API. Fixed (`packages/connectors/src/sdk/html-to-text.ts` now decodes entities first, then strips tags) and directly asserted in `greenhouse/index.test.ts` (`descriptionMd` must not contain `&lt;`).
- **Lever**: `GET https://api.lever.co/v0/postings/palantir?mode=json` (2 of 273 real Palantir postings; netflix/lever/reddit/figma/attentive/ramp were probed first and returned 0 postings or 404).
- **Ashby**: `GET https://api.ashbyhq.com/posting-api/job-board/notion` (2 of 141 real Notion postings). This recording disproved an assumption in the first draft: no live board (Notion/Ramp/Linear/OpenAI/Substack, all probed) ever returns a populated `compensation` field, and `isRemote` can be `null`, not just boolean — both now reflected in the fixture/connector/tests instead of an invented shape.
- **RSS**: `GET https://weworkremotely.com/categories/remote-programming-jobs.rss` (2 of ~30 real items). This recording surfaced a second real bug: `fast-xml-parser`'s default entity-expansion cap (1000) is too low for a real multi-item job feed with heavily-formatted HTML descriptions — the live fetch threw `Entity expansion limit exceeded: 1104 > 1000` before any code review would have caught it. Fixed by raising (not disabling) the cap via the `processEntities` option.
- **USAJobs — the one exception, hand-authored, disclosed as such**: USAJobs requires a registered (free, but account-gated) `Authorization-Key` this task does not have. What WAS verified live: an unauthenticated/invalid-key call to the real endpoint returns HTTP 401 with an RFC-9110 problem-details body, which is exactly what `usajobs/index.test.ts`'s `auth_error` test asserts against. The success-path fixture is hand-authored to match USAJobs' published API reference shape as closely as possible — this gap is flagged in the test file's own header comment, not silently presented as a live recording.

**Test results** (isolated `node:22-slim` container, `pnpm install`, no live network calls in the default run):
- `vitest run --config vitest.config.ts packages/connectors` → **40/40 tests pass** (greenhouse 6, lever 4, ashby 4, usajobs 5, rss 5, manual 4, sdk/contract-test-kit 4, sdk/registry 3, sdk/html-to-text 5).
- Full monorepo unit suite: **219/219 passing** (up from 218 after task 027 + the new `html-to-text.test.ts`).
- `turbo run typecheck` across all 10 workspace packages (including `apps/api`/`apps/worker`/`apps/web`, unaffected by this task) → all 10 succeed.
- `eslint .` (whole repo) → 0 errors, 6 warnings (all pre-existing-pattern `no-explicit-any` in the registry's heterogeneous `Map`, plus one in the RSS Atom-entry mapper — same style as task 026's registry).
- `node scripts/verify-boundary-enforcement.mjs` → both layers still catch violations.
- **Nightly live canary actually run** (`vitest run --config vitest.nightly.config.ts packages/connectors`, real network, not simulated): Greenhouse, Lever, Ashby, and RSS each genuinely fetched ≥1 real job from the real live endpoint in this run; USAJobs correctly reported "skipped: no key configured" (its designed behavior, not a failure). `.github/workflows/nightly.yml` gained a `connectors-live-canary` job (`continue-on-error: true`, non-blocking) that runs this nightly against the real internet, with `USAJOBS_API_KEY`/`USAJOBS_USER_AGENT_EMAIL` as optional secrets the user can add later.
- Manual connector: acceptance criterion "re-expressed as a `ConnectorPort` implementation — not new functionality" — verified by inspection: `create-manual-job.ts` (task 011) is untouched; `manual/index.ts` only adds a `ConnectorPort` wrapper whose `fetchJobs` always yields nothing (manual postings are pushed via the existing use case, never pulled by a scheduler) and whose `key`/`complianceClass` match `job_postings.source_connector_key`'s existing `'manual'` default.

### Known gap
USAJobs' success-path fixture is representative/hand-authored, not live-recorded (see above) — the connector's request construction and error-handling ARE verified against the real endpoint (real 401 response), just not a real successful search result. Fully closeable once a `USAJOBS_API_KEY` + `USAJOBS_USER_AGENT_EMAIL` secret pair is available; the nightly job is already wired to pick them up.
