# 022 — API: profile & document routes

**Status:** DONE | **Milestone:** M3

## Objective
Fastify routes exposing profile and document CRUD, matching task 011's auth/route conventions (session-authenticated, scoped to the requesting user).

## Files affected
- `apps/api/src/routes/profile.ts` (`GET/PUT /api/profile`, `POST /api/profile/sections`)
- `apps/api/src/routes/documents.ts` (`GET /api/documents`, `POST /api/documents`, `POST /api/documents/:id/versions`, `GET /api/documents/:id/versions/:versionId`)
- route registration in `apps/api/src/app.ts`

## Dependencies
020, 021, 011

## Acceptance criteria
- [x] All routes require an authenticated session; requests for another user's profile/document return 404 (not 403 — avoid confirming existence, matching existing auth posture)
- [x] Request/response bodies validated against the task 020 zod contracts
- [x] `audit_log` entry written on document version creation (design §2: "Written for: ... exports" — document creation is an adjacent auditable event)

## Test plan
Integration: spin up the Fastify app against real Postgres, exercise each route including the cross-user 404 case.

## Deliberate deviation from the task's literal path strings
The task text writes `/api/profile`, `/api/documents`, etc. No other M2
route in this codebase is prefixed with `/api` — `/jobs`, `/applications`,
`/board`, `/auth/*` are all unprefixed (`apps/api/src/routes/*.ts`). Per the
task's own Objective ("matching task 011's auth/route conventions"), the
established convention wins over the literal path strings: these routes are
`/profile`, `/profile/sections`, `/documents`, `/documents/:id`,
`/documents/:id/versions`, `/documents/:id/versions/:versionId` — same
pattern task 011 already used for a deliberate deviation
(`tasks/011.md`'s cross-owner-404-not-403 note).

Also added `GET /documents/:id` (not in the task's literal route list) —
the natural single-document-detail counterpart to `GET /jobs/:id` (task
011), needed by task 025's document detail/history view.

## Test plan actually executed

`registerProfileRoutes` (`apps/api/src/routes/profile.ts`):
`GET /profile` (404 before any profile exists — distinguishes "no profile
yet" from "empty profile", per `get-profile.ts`'s doc comment),
`PUT /profile` (upsert: tries `updateProfile` first, falls back to
`createProfile` on `not_found` — no business logic in the route, just two
single-purpose use case calls composed with a status-code branch: 200 on
update, 201 on create), `POST /profile/sections`.

`registerDocumentRoutes` (`apps/api/src/routes/documents.ts`):
`GET /documents`, `POST /documents` (201 — unlike `POST /jobs`'s 202, there
is no async work pending after creation), `GET /documents/:id`,
`POST /documents/:id/versions`, `GET /documents/:id/versions/:versionId`.
Every route's ownership check flows through `findByIdForUser`/
`findActiveForUser`, which return `null` (→ `not_found` → HTTP 404) for a
resource owned by someone else — same posture as task 011's job/application
routes, verified by two dedicated cross-owner tests.

`AppDeps` (`apps/api/src/app.ts`) extended with `profiles: ProfileRepository`
and `documents: DocumentRepository`; `apps/api/src/main.ts` wires
`DrizzleProfileRepository`/`DrizzleDocumentRepository`.

Verified against real Postgres + Redis in an isolated Docker setup: the
`m3pg` (pgvector/pg16) and `m3redis` containers from task 021, plus the
`node:22-slim` runner container on the same `m3net` network:
- `vitest run --config vitest.integration.config.ts apps/api` →
  **28/28 tests pass** (12 new: profile GET/PUT-upsert/sections + ownership,
  documents CRUD/versions/audit/ownership, malformed-payload 400s; all 16
  pre-existing M2 API tests — auth, jobs, applications/board, ops surface,
  unmatched-route — still green, unaffected by the `AppDeps` extension).
  Notably `POST /documents/:id/versions` → `audit_log` row assertion queries
  the real table directly and confirms `action = 'document.version_created'`
  and `subject_id = documentId`.
- `vitest run --config vitest.integration.config.ts packages/infrastructure`
