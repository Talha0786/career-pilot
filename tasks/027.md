## Acceptance criteria
- [ ] `connector_configs.credentials_ref` stores a reference (e.g. env var name or secrets-manager key), never the raw credential/API key in the row — matches the security model's posture on stored credentials
- [ ] `job_postings` gets a unique index on `(source_connector_key, external_id)` per design §2, added without breaking existing rows (default `external_id` backfill strategy documented in the migration)

### Design notes / deviations from the literal task scope

- **Domain layer extended too**, beyond the "Files affected" list (schema/migration/repositories only): `packages/domain/src/discovery/job-posting.ts` gained `PostingLocation`/`RemoteType`/`PostingSalary`/`PostingStatus` types, new snapshot fields, a `JobPosting.ingest(...)` factory (parallel to `createManual`, used by connector-sourced postings), and `assignDedupGroup`/`markClosed`/`markExpired` methods. This was necessary, not optional scope creep: without an aggregate-level way to construct/mutate these fields, the new DB columns would be write-only from raw SQL and tasks 028-031 would have nowhere to put normalized `RawJob` data through the domain layer. Documented here per the "new work gets a new task file" rule's spirit — this is a direct, unavoidable corollary of the schema task's stated objective ("the job_postings columns M2 deliberately left out"), not an unrelated widening.
- **Application ports extended too**: `ConnectorConfigRepository`/`IngestionRunRepository`/`ConnectorConfig`/`IngestionRun` types added to `packages/application/src/ports/repositories.ts`, and `JobPostingRepository` gained `findBySourceAndExternalId` — required so the infrastructure repositories (task 027's actual file list) have an application-defined port to implement, per the Clean Architecture rule this whole codebase enforces in CI.
- **Backfill strategy for `external_id`**: no backfill SQL. The column is nullable; Postgres unique indexes treat every `NULL` as distinct from every other `NULL`, so existing rows (all `external_id IS NULL`) never collide with each other or with future non-null connector-assigned values. Verified directly by test 5 above, not just asserted in a comment.
- **Migration numbering**: named `0003_connectors.sql` per the task's literal instruction, applied directly on top of `0001_init.sql`. `0002_*` (M3's profile/document migration) does not exist on this branch — M3 and M4 are documented as independent parallel tracks (`tasks/README.md`) that merge later. The migration is self-contained (references no M3 table), so it is safe both now (0001→0003) and after merge (0001→0002→0003, applied in numeric order by `migrate.ts`'s filename sort). Flagged explicitly rather than silently hoping the numbers reconcile themselves.
