Notable design decisions, documented inline in both the schema file and the
SQL migration:
- `content_text` on `profile_sections` is populated by the repository from
  `ProfileSection.toContentText()` (domain, task 019); `content_tsv` is a
  REAL Postgres `GENERATED ALWAYS AS (to_tsvector(...)) STORED` column
  derived from it, with a GIN index â€” FTS can't drift out of sync with
  `content_text` by construction (design Â§2 rationale). `content_tsv` isn't
  in the Drizzle table definition since nothing queries it yet.
- `career_profiles_user_active_unique`: a partial unique index on
  `(user_id) WHERE is_active` â€” defense-in-depth in the DB for the
  application-layer "one active profile per user" policy from task 020's
  `createProfile`, closing the race a plain read-check-write leaves open.
- `documents.current_version_id` has **no FK constraint** â€” it would be
  circular with `document_versions.document_id`. Documented as a deliberate
  simplification; integrity is enforced at the application layer instead
  (only ever set to a version id inserted in the same `save()` call).
- `document_versions`: no `updated_at` column, unique index on
  `(document_id, version)` (duplicate version numbers are a hard DB error,
  not just convention), and the repository's `save()` only ever `INSERT`s
  new version rows â€” the one `onConflictDoUpdate` on this table exists
  purely to let `renderedPdfKey` (task 024) be filled in post-hoc on an
  already-persisted version; content/version/source/createdAt are never in
  its `set` clause.
- Added `drizzle.config.ts` at the repo root (didn't exist before, even
  though `package.json`'s `db:generate` script already referenced it) so
  `drizzle-kit check` is actually runnable â€” this repo's migrations are
  hand-written SQL applied by `migrate.ts` (task 013), not drizzle-kit's own
  migrator; `drizzle-kit check` is used purely as a schema-drift detector.

**Bug found by the integration tests, fixed in task 019's code**:
`CareerProfile.factsHash`, computed pre-save, did NOT match the same
profile's `factsHash` after `save()` â†’ `findByIdForUser()` round-tripped it
through Postgres. Root cause: Postgres `jsonb` does not preserve object key
insertion order (it re-encodes to its own internal order), so
`JSON.stringify` over the same logical content before and after a jsonb
round-trip can produce different strings. Fixed by adding a `stableStringify`
helper (recursively sorts object keys, preserves array order) in
`career-profile.ts` and using it in `computeProfileFactsHash`. Added a unit
test (`career-profile.test.ts`, "is independent of object key order within
content") reproducing the exact failure mode, plus the integration test that
originally caught it stays in the suite as a regression guard.
