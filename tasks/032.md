## Test plan actually executed

**`e2e/chaos/connector-failure.spec.ts` — the primary proof, run for real, actual transitions observed:**

- A dedicated test-only worker entry (`apps/worker/src/chaos-connector-worker.ts`, not used by `main.ts`/docker-compose) is spawned as a REAL OS process via `tsx`, registering a `chaos-broken` connector whose `fetchJobs` genuinely `throw`s an `Error` (not a typed `Result`, not a flag) on its first N real invocations, and a `chaos-healthy` connector that always succeeds — both real `ConnectorPort` implementations running through the real `runConnectorIngestionOnce`/`createRunConnectorIngestionWorker`/`makeUpdateConnectorHealthUseCase` machinery, consuming a real BullMQ queue against real Postgres + Redis.
- Observed, in order, against the real database (not asserted-into-existence): 3 real failed `ingestion_runs` rows for `chaos-broken` (`error: "Connector threw unexpectedly: Error: chaos: deliberate failure #1/2/3 of 3"`) interleaved with 3 real `ok` rows for `chaos-healthy` — `chaos-broken.health` flips from `healthy` → `degraded` at exactly `consecutiveFailures = 3` (`DEGRADED_AFTER_CONSECUTIVE_FAILURES`), while `chaos-healthy.health` stays `healthy` throughout with `consecutiveFailures = 0` the whole time and all 3 of its jobs actually landed in `job_postings` — **isolation, proven, not assumed**.
- A 4th real run of `chaos-broken` (past its fail count) succeeds for real — `health` flips back to `healthy`, `consecutiveFailures` resets to 0, `lastSuccessAt` is stamped. This is the literal "flips back to healthy after a subsequent success" acceptance criterion.
- Test passed twice in isolation and once more running alongside task 014's `worker-kill.spec.ts` in the same `vitest.e2e.config.ts` suite (2/2 files, 2/2 tests, both real chaos tests coexisting cleanly).

**A real concurrency bug was found and fixed while building this test, not glossed over:** the first implementation of health tracking did `findById` then `save` as two separate calls (mirroring the rest of this codebase's repository style). The chaos test's own interleaved broken/healthy runs exposed a **lost-update race**: 3 real, correctly-recorded `ingestion_runs` failures only advanced `connector_configs.consecutive_failures` to 2, because two concurrent `updateConnectorHealth` calls for the same connector both read the same stale counter value before either wrote back — the exact class of bug task 015/016 already closed for budget spend, just for a different counter that hadn't been protected yet. Fixed by making `ConnectorConfigRepository.recordRunOutcome` a single atomic `UPDATE ... SET consecutive_failures = consecutive_failures + 1 ... RETURNING` statement (Postgres' row-level lock serializes concurrent updates to the same row), replacing the two-step read/write in `update-connector-health.ts`. Re-ran the chaos test after the fix and confirmed the counter now advances correctly under real concurrent load.

### A note on process cleanup during verification (not a code defect)
While debugging, `ps`-equivalent inspection of the verification container found that a killed chaos-worker's `tsx`-spawned Node process can occasionally linger as an orphan after `child.kill('SIGKILL')` returns, when tests are re-run manually many times against one long-lived container. This is a pre-existing characteristic of task 014's own `worker-kill.spec.ts` (confirmed: it does the exact same thing, orphaning up to 2 processes per run) — inherent to the tsx-spawn-and-SIGKILL pattern this task was instructed to match, not something task 032 introduced. It has zero effect on real CI (each job runs in a fresh, ephemeral environment that's discarded after the job) and only became visible here because of this session's practice of reusing one Docker container across many manual verification passes.
