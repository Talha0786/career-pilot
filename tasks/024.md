# 024 — Document engine: structured model → PDF/DOCX rendering

**Status:** DONE | **Milestone:** M3

## Objective
Render a `DocumentVersion`'s structured content into PDF and DOCX. Constrained to **2 templates** (roadmap's explicit scope guard against the "doc rendering rabbit hole").

## Files affected
- `packages/infrastructure/src/documents/render/{template-classic,template-modern}.ts`
- `packages/infrastructure/src/documents/render/pdf-renderer.ts`, `docx-renderer.ts`
- `packages/application/src/documents/commands/render-document.ts` (writes `rendered_pdf_key` per design §2, stores the artifact — reuse whatever object-storage approach the compose stack already provides, or local volume + signed API route if none exists yet)
- golden-file fixtures under `packages/infrastructure/test/fixtures/documents/`

## Dependencies
019, 021

## Acceptance criteria
- [x] Round-trip: structured content → PDF renders correctly against a golden-file test (visual or text-extraction diff, whichever is more stable in CI)
- [x] Same for DOCX
- [x] Rendering is deterministic (same input → byte-identical or diff-stable output) so golden-file tests don't flake
- [x] Exactly 2 templates shipped — no speculative template system

## Test plan
Golden-file tests in CI (extend `packages/infrastructure/test`), run against both templates.

## Test plan actually executed

**Renderers**: `packages/infrastructure/src/documents/render/{pdf-renderer,docx-renderer}.ts` render all three `DocumentContent` kinds (resume/cover_letter/other — task 019's domain model) via `pdfkit` (PDF) and `docx` (DOCX), dispatched through a composite `DocumentRenderer` implementing the new `DocumentRendererPort`. Exactly 2 templates (`template-classic.ts`, `template-modern.ts` — font family + size choices only, `render-style.ts` holds the shared `RenderStyle` shape and `styleFor()` resolver); no template registry or plugin system.

**Golden-file tests are TEXT-EXTRACTION diffs, not byte diffs** — the acceptance criterion explicitly allows this ("visual or text-extraction diff, whichever is more stable in CI"), and it turned out to be necessary, not just a preference: `pdfkit` embeds a random trailer `/ID` per render (`crypto.randomBytes`, independent of content or a pinned `CreationDate`), so raw PDF bytes are NOT byte-identical across runs even with every controllable metadata field pinned; this `docx` version's `IPropertiesOptions` doesn't even expose `created`/`modified` to pin. `document-render.test.ts` renders → extracts text back via the task 023 extractors (`extractPdfText`/`extractDocxText` — real reuse, not a parallel code path) → diffs against a committed fixture. 6 golden fixtures under `packages/infrastructure/test/fixtures/documents/` (resume × {classic,modern} × {pdf,docx}, cover-letter × one template × each format), each hand-inspected before being frozen (not blindly accepted from a first run) — confirmed correct section ordering, bullet rendering (`•` in PDF), contact-line formatting, and heading casing for both templates.

**Determinism, honestly scoped**: two explicit tests assert **extraction-stable** output — same input rendered twice produces byte-different files (proven, not glossed over) but identical extracted text — for both PDF and DOCX. This is the acceptance criterion's "diff-stable" branch, exercised for real rather than assumed.

**`render-document.ts`** (application command): loads an existing `DocumentVersion` by id (ownership-scoped, same `findByIdForUser` posture as every other route), renders via the port, stores the bytes via a new `ObjectStoragePort`, and calls `Document.attachRenderedArtifact` — the one legitimate post-hoc field on an append-only version (task 019). Never touches content/version/source/createdAt; verified explicitly (a dedicated API test renders a version twice and confirms no new version row is created).

**Storage — "local volume + signed API route" per the task's own fallback guidance**: no object-storage service exists in the compose stack (checked `docker-compose.yml` — no MinIO/S3), so `LocalFileObjectStorage` (new) writes under a configurable directory (`DOCUMENT_STORAGE_DIR`, default `./data/documents`), served through an auth-gated `GET /documents/:id/versions/:versionId/download` route rather than a cryptographically presigned URL (no S3-compatible presign capability to generate one from) — documented as the practical interpretation in `object-storage.port.ts`'s docstring. `POST /documents/:id/versions/:versionId/render` triggers rendering.
