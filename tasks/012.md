## Architecture notes
- **Dev-time single-origin proxy**, not CORS: `next.config.ts` rewrites `/api/*` and `/ws` to the API origin. The browser only ever talks to the Next.js origin, so the session cookie's `SameSite=Lax` never crosses an origin boundary and no `credentials:'include'` gymnastics are needed. Prod does the same thing for real via Caddy (task 013).
- **Webpack extensionAlias fix**: workspace packages (e.g. `@careerpilot/contracts`) use NodeNext-style explicit `.js` import specifiers that resolve to `.ts` source. `tsc`/`tsx` handle this natively; webpack (Next's bundler) doesn't by default — added `config.resolve.extensionAlias = { '.js': ['.ts', '.tsx', '.js'] }` to `next.config.ts`.
- **Drag-and-drop** uses the native HTML5 DnD API (`draggable`, `dragstart`/`dragover`/`drop`) — no library. Optimistic move on drop, reverted (refetch) on a rejected transition (e.g. illegal stage move → 409).

## Bugs found and fixed while verifying live in a browser
1. **`e.currentTarget` read after `await` in the paste-job submit handler.** React nulls out a synthetic event's `currentTarget` once the synchronous dispatch phase ends, so `e.currentTarget.reset()` — called after `await api.createJob(...)` — threw `TypeError: Cannot read properties of null`, which the `catch` block turned into a false "Could not create job" error **even though the job was created successfully** (confirmed via network log: every request had already succeeded). Reproduced twice live before diagnosing it. Fixed by capturing `const formEl = e.currentTarget` synchronously before the first `await`.
2. **Embedding failures could spin forever in the UI.** `job-posted.handler.ts` only published a `ws:job.embedded` "failed" event for `budget_exceeded`; for retryable provider errors (`provider_unavailable`, `rate_limited`) it logged and threw for BullMQ to retry, publishing nothing — so if retries exhausted, the DB correctly ended up `embedding_status='failed'` but the browser was never told and the card stayed on "Embedding…" indefinitely. Fixed by publishing the WS event unconditionally on every failed attempt (a later successful retry simply supersedes it with a 'ready' push). Verified live by pointing the worker at an unreachable LLM endpoint and watching the card flip to a red "Embedding failed" state with no user action.
