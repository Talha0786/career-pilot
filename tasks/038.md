# 038 — Match rubric scoring pipeline

**Status:** TODO | **Milestone:** M5

## Objective
Implement the match-scoring pipeline from `docs/06-agent-design.md` §3 ("Match rubric scoring | pipeline | mid | ScoreComponents schema | embedding prefilter caps volume"): embedding-based ANN prefilter (task 036) narrows candidates, then a guarded LLM call scores each candidate against a rubric, persisted and surfaced on the board.

## Files affected
- `packages/contracts/src/matching.ts` (new) — `ScoreComponentsSchema` (e.g. `{ skillsMatch: 0-1, experienceMatch: 0-1, seniorityFit: 0-1, overall: 0-1, rationale: string }` — check `docs/01-system-design.md`/`docs/02-database-design.md` for any prior rubric-dimension spec before inventing dimensions from scratch), `MatchScoreDtoSchema`.
- `packages/domain/src/matching/match-score.ts` (new) — `MatchScore` value object/aggregate: `{ id, profileId, jobPostingId, components: ScoreComponents, computedAt, factsHash /* profile factsHash at scoring time, for staleness */, embeddingModel }`.
- `packages/application/src/ports/repositories.ts` — add `MatchScoreRepository` port.
- `packages/infrastructure/src/db/migrations/0005_match_scores.sql` (new) — `match_scores` table, unique on `(profile_id, job_posting_id)`, upsert-on-recompute.
- `packages/infrastructure/src/db/repositories/drizzle-match-score.repository.ts` (new).
- `packages/application/src/matching/commands/score-match.ts` (new) — orchestrates: ANN prefilter (036) → for each candidate, load `prompts/match-score/v1.md` (034) → guarded `llm.complete()` → validate against `ScoreComponentsSchema` (zod; one repair-attempt on validation failure per ADR-006) → persist via `MatchScoreRepository`.
- `apps/worker/src/main.ts` — wire a handler (job-triggered, e.g. on profile embed completion or on-demand) to run `score-match` for a profile against its ANN-prefiltered candidate set.
- `apps/api/src/app.ts` — first place `AppDeps` needs a `GuardedLlmPort` + `MatchScoreRepository`; add `apps/api/src/routes/matching.ts` with an on-demand `POST /profiles/:id/rescan` or similar (check `docs/01-system-design.md` for any existing route-naming convention for this feature before inventing one).
- `packages/contracts/src/board.ts` — extend `ApplicationCardSchema` (or the job-posting-facing DTO) with an optional `matchScore` field so the board can surface it.
- Tests: unit (packages/application, fake LLM + fake repos), integration (real Postgres — persist/query/upsert), API integration test for the new route.

## Dependencies
- 033 (real CostEstimator — this is the first production `complete()` call site, budget accuracy matters).
- 034 (PromptStore).
- 035 (profile embeddings must exist to prefilter against).
- 036 (ANN prefilter).

## Acceptance criteria
- Given a profile and a set of embedded job postings, `score-match` returns rubric scores for the ANN-prefiltered top-N only (not every job posting — this is the whole point of the prefilter, keep it enforced with a test, not just a comment).
- Invalid/malformed LLM JSON output triggers exactly one repair retry, then a typed failure (not a crash) — matches the ADR-006 "structured outputs validated with zod; on validation failure → one repair attempt → fail the job with raw output stored for triage" rule verbatim.
- Scores are queryable per profile and surface on the board DTO.
- Budget guard is exercised on every call (no bypass path).

## Test plan

- Integration: real Postgres persistence + upsert-on-recompute.
- API integration: rescan route round-trip.
