# 040 — Claim verification pass (anti-hallucination gate)

**Status:** TODO | **Milestone:** M5

## Objective
Implement `docs/06-agent-design.md` §4 point 2-3: a **separate call, different prompt, adversarial framing** that extracts every factual claim from a tailored draft and maps each to a fact ID or marks it `UNSUPPORTED`; unsupported claims trigger ≤2 targeted regeneration retries, then `needs_human` with flagged claims. This is the mandatory safety gate — task 039's structural `supportingFactIds` check is a cheap first pass, this is the real adversarial check the roadmap's acceptance criteria and eval harness (042) are gated on ("regression gate: 0 unsupported claims surviving verification").

## Files affected
- `prompts/verify-claims/v1.md` (fill in real content — stubbed in 034) — adversarial framing per docs/06 §4: given the fact list + the generated draft, extract every claim and independently re-derive its fact mapping (must NOT simply echo back the draft's own `supportingFactIds` claims — that would make this pass useless as an adversarial check; it must re-derive from the draft text + fact list only).
- `packages/contracts/src/tailoring.ts` (extend) — `ClaimAuditSchema { claims: [{ text: string, factId: string|null /* null = UNSUPPORTED */, confidence: number }] }`.
- `packages/application/src/tailoring/commands/verify-claims.ts` (new) — guarded `llm.complete()` call, zod-validate `ClaimAuditSchema`.
- `packages/application/src/tailoring/commands/tailor-document.ts` (from 039, extend) — after generation, run `verify-claims`; if any claim is `UNSUPPORTED`, retry generation with the unsupported claims fed back into the prompt as "avoid this" hints (≤2 attempts); if still unresolved after retries, persist the document version with a `needs_human: true` flag (extend `DocumentVersion` if it doesn't already have a review-status field) and the flagged claims attached for the review UI (041) to render.
- `packages/domain/src/documents/document.ts` / `document-version.ts` — add whatever minimal state is needed to represent "generated but pending human review with N flagged claims" if not already expressible.
- Tests: unit (fake LLM sequences: clean-first-try, unsupported-then-clean-on-retry, unsupported-through-all-retries → needs_human), integration (real Postgres, persisted `needs_human` state + flagged claims retrievable by the API for the review UI).

## Dependencies
- 039 (tailoring pipeline produces the draft this verifies).
- 034 (PromptStore).

## Acceptance criteria
- Claim verification is a genuinely separate LLM call with its own prompt file, not a reuse/rerun of the tailoring prompt.
- 0-unsupported-claims is enforced as a hard gate before a document version can be marked ready for export — `needs_human: true` documents are not exportable until a human resolves the flagged claims (export-blocking check enforced in code, test it explicitly — don't just trust the UI to hide the button).
- Retry budget is exactly ≤2 regeneration attempts, then stop and flag — proven with a fake-LLM test that always returns unsupported claims (must terminate after 2 retries, not loop or hang).

## Test plan
- Unit: the three retry-outcome paths above with a fake LLM.
- Integration: real Postgres — `needs_human` documents are queryable, export route (extend/verify against task 022's document routes) rejects export of a `needs_human: true` version with a clear error, not a silent no-op.
