# 035 — Profile embedding use case

**Status:** TODO | **Milestone:** M5

## Objective
`CareerProfile` already has `embeddingStatus/embeddingModel/embedding/embeddedFactsHash` fields and an `isEmbeddingStale` getter (task 019), but nothing ever calls `llm.embed()` for a profile — only `embed-job-posting.ts` exists. Add the analogous use case and wire it into the worker so profiles get embedded (and re-embedded on `factsHash` drift) the same way job postings already do.

## Files affected
- `packages/application/src/profile/commands/embed-career-profile.ts` (new) — mirror `packages/application/src/discovery/commands/embed-job-posting.ts`'s shape exactly (idempotent by model, guarded LLM call, persists via `ProfileRepository`).
- `packages/domain/src/profile/career-profile.ts` — reuse existing `attachEmbedding()`/`isEmbeddingStale` (should need no changes if task 019 built them correctly; verify and only touch if there's a real gap).
- Event wiring: profile creation/update needs to enqueue an embed job the same way `job-posted.handler.ts` does for job postings — find the equivalent profile domain event (check `packages/domain/src/profile` for a `ProfileUpdated`/`ProfileSectionsChanged` event; if none exists, add one following the existing `DomainEvent` pattern and wire it through the outbox, same transactional-outbox discipline as ADR-007).
- `apps/worker/src/main.ts` — register the new handler alongside the existing job-posting embed handler.
- `packages/application/test/unit/embed-career-profile.test.ts` (new).
- `packages/infrastructure/test/integration/` — add a real-Postgres integration test proving profile embed idempotency (re-running on an unchanged `factsHash` is a no-op; changed `factsHash` triggers re-embed), mirroring whatever integration test already covers `embed-job-posting`.

## Dependencies
None (uses only already-DONE M2/M3 machinery: `GuardedLlmPort`, `CareerProfile`, outbox).

## Acceptance criteria
- Creating/updating a profile eventually results in `embeddingStatus: 'ready'` with a populated `embedding` vector, via the same outbox → worker → guarded-LLM path as job postings.
- Re-embedding is idempotent: unchanged `factsHash` does not re-trigger a paid embed call.
- `isEmbeddingStale` flips correctly when sections change (should already work from task 019 — this task's test plan must prove it end-to-end through the new use case, not just re-assert the M3 unit test).

## Test plan
- Unit: use case with a fake `LlmPort` + fake `ProfileRepository`, embed happy path, idempotent-skip path.
- Integration: real Postgres, real outbox relay, real worker handler — profile create → embed job flows through and lands `embeddingStatus: 'ready'`.
