# 015 — Atomic budget increment (concurrency gap found in testing)

**Status:** DONE (primitive proven) — wiring into GuardedLlmPort tracked as 016
**Milestone:** M2

## Objective
Close the race condition found while testing task 009: the budget guard's
pre-dispatch check is read-then-decide, so two concurrent requests for the
same user can both read spend under the cap and both pass, over-spending.

## What was built and proven
- `PostgresBudgetStore.withUserBudgetLock()` — `pg_advisory_xact_lock(hashtext(userId))`
  serializes the check-then-write per user without requiring every caller to
  share one DB transaction.
- **Proven twice, against real Postgres, 5 consecutive clean runs:**
  1. `budget-lock.test.ts` "WITHOUT the lock" — reproduces the bug for real:
     naive read-then-write lets more than 3 of 20 concurrent callers through
     against a 3-call budget.
  2. `budget-lock.test.ts` "WITH pg_advisory_xact_lock" — exactly 3 of 20
     succeed, every run.
- **Real bug found and fixed along the way**: the first version of the "with
  lock" test used a razor-thin budget boundary (`0.0001 * 3` compared via JS
  floating point) and failed even with the lock working correctly — IEEE 754
  rounding, not a logic error. Fixed by giving the boundary slack. Lesson
  documented in the test: money comparisons belong in SQL NUMERIC, not
  JS float equality at an exact boundary.

## Remaining (tracked as task 016)
`GuardedLlmPort.embed()` does not yet call `withUserBudgetLock` — the
primitive exists and is proven correct in isolation, but isn't wired into the
guard's actual call path yet. That wiring plus an integration test at the
guard level (not just the store level) is task 016.

## Test plan (satisfied)
Integration (real Postgres, no mocks): concurrent connections racing the same
user's budget; exact-count assertions, not "at least/most"; re-run 5x to rule
out flakiness.
