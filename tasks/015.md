# 015 — Atomic budget increment (concurrency gap found in testing)

**Status:** TODO | **Milestone:** M2

## Objective
Close a real race condition found while testing task 009: the budget guard's
pre-dispatch check is read-then-decide. Two concurrent requests for the same
user can both read spend=0 against a near-exhausted budget and both pass,
over-spending past the cap. The in-memory test fake (`InMemoryBudgetStore`)
reproduces this honestly rather than hiding it.

## Files affected
- `packages/infrastructure/src/llm/postgres-budget-store.ts`

## Dependencies
006 (schema), 009 (LLM port)

## Acceptance criteria
- [ ] Spend check + increment happens as a single atomic SQL statement
      (`UPDATE ... SET spend = spend + $1 WHERE spend + $1 <= $budget RETURNING *`,
      or an explicit row lock) — not a separate read then a separate write
- [ ] A concurrency test with N parallel embed calls against a tight budget
      allows at most floor(budget/cost) through, never more
- [ ] Single-caller behavior (the common case) is unchanged

## Test plan
Integration (real Postgres): fire 20 concurrent requests at a budget sized
for exactly 3; assert exactly 3 succeed and 17 are `budget_exceeded`, every
run, not just on average.
