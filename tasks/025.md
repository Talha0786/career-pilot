## Objective

## Acceptance criteria
- [x] Import confirm screen shows parsed fields with confidence/diff markers so low-confidence fields are obviously reviewable (supports the accuracy story from 023 ‚Äî errors must be cheap to catch, per roadmap risk mitigation)
- [x] Document list shows a visible "stale" indicator when a document's `profile_facts_hash` no longer matches the current profile
- [x] No client-side state library added beyond what 012/018 already established

## Test plan
Manual verification in the browser (per project convention for UI work ‚Äî start the dev server, exercise upload‚Üíconfirm, edit a profile section, verify staleness badge appears after a profile edit). Playwright e2e for the golden path only (upload a fixture resume ‚Üí confirm ‚Üí see it on `/profile`).

## Deliberate deviations
- **File locations**: `apps/web/src/app/...` and `apps/web/src/lib/api/...`, not `apps/web/app/...` / `apps/web/lib/api/...` as the task text writes. The existing app (task 012) already uses the `src/app` App Router layout (`apps/web/src/app/{board,login,register}/page.tsx`) ‚Äî following that established convention over the task's literal paths, same reasoning as tasks 022/023's path-prefix deviations.
- **State management**: no state library was added ‚Äî every page uses plain `useState`/`useEffect`, matching `board/page.tsx`'s existing pattern exactly (acceptance criterion satisfied by omission).

## Test plan actually executed

**Pages** (`apps/web/src/app/{profile,profile/import,documents}/page.tsx`) and typed clients (`apps/web/src/lib/api/{profile,documents}.ts`, plus `export`ing the existing `request` helper from `api-client.ts` rather than duplicating a fetch wrapper). `Board`'s header gained Profile/Documents nav links; the new pages link back to Board/Profile/Documents for cross-navigation, matching the existing header pattern.

- `/profile`: shows a create form when no profile exists (404-driven empty state), otherwise a details-edit form (title/summary, with the derived `isEmbeddingStale` badge) plus a section list and an "Add section" form whose fields adapt to the selected `ProfileSectionKind` (experience/education/project/skill_group/certification/summary), each producing the exact `AddSectionRequestSchema` discriminated-union shape.
- `/profile/import`: upload form ‚Üí polls `GET /profile/import/:draftId` every 1.2s while `status === 'processing'` ‚Üí renders `failed` with the typed error message, or `ready` with contact fields, summary, and every parsed section EACH tagged with a confidence badge (`high confidence` ‚â•0.7 green / `review` ‚â•0.4 amber / `low confidence` red) and a checkbox to exclude it from confirmation ‚Äî the literal acceptance criterion.
- `/documents`: table of documents with a `stale`/`up to date` `Badge` driven directly by the API's `isStale` field (list-documents.ts, task 020); expandable rows show version history with per-version Render PDF/DOCX buttons that become a real Download link once rendered.

**This was verified LIVE against the real running stack**, not just typecheck/lint ‚Äî a genuine, non-negotiable step given how much of this session's real bugs (tasks 023/024) were caught exactly this way. Ran `api` + `worker` + `next dev` (web) as real processes inside an isolated Docker container network (joined to the same `m3pg`/`m3redis` containers used throughout this session), with ports published to the host, and drove the actual app through the Browser tool:

1. Registered a new user, logged in, redirected to `/board` ‚Äî nav links to Profile/Documents present and working.
2. `/profile` with no profile yet ‚Üí create form ‚Üí submitted ‚Üí profile created, editor now shows "Embedding: stale" (correct ‚Äî nothing embedded).
3. Added an Experience section via the dynamic Add-Section form (Senior Software Engineer @ Acme Corp, dated, with bullets) ‚Üí appeared correctly in the section list.
4. `/documents` empty state ‚Üí created a new "resume" kind document ‚Üí expanded the row ‚Üí Add-Version form (kind-adaptive, same pattern as Add-Section) ‚Üí added a real resume version.
5. Clicked "Render PDF" ‚Üí UI swapped to a "Download PDF" link ‚Üí fetched it: **200, `content-type: application/pdf`, bytes start with `%PDF`** (real PDF, not a stub).
6. Added a second version, clicked "Render DOCX" ‚Üí **200, `content-type` is the DOCX mimetype, bytes start with `PK`** (real zip/DOCX).
7. **Resume import, failed path**: uploaded a deliberately-corrupt "PDF" through the real file input (via a programmatically-constructed `File`/`DataTransfer`, since this tool stack has no native OS file-picker automation) ‚Üí real `POST /profile/import` ‚Üí real worker ‚Üí real `pdfjs-dist` extraction failure ‚Üí UI correctly rendered "Could not parse this file: Could not parse PDF: Invalid PDF structure." ‚Äî the exact typed-error path from task 023, visible in the actual UI.
8. **Resume import, ready path**: uploaded again to get a real `draftId` from the real API; briefly stopped the worker process so the draft stayed `processing`, then wrote a realistic `ResumeImportDraftRecord` directly to the same Redis the API reads from (same technique the task 023 API integration tests already use) so the polling UI would observe a `ready` transition without racing the real worker. The review screen rendered exactly as designed: Name/Email/Phone each with their own confidence badge (including "not found" + `low confidence` for the missing phone), Summary with its badge, and two sections (`Staff Engineer` ‚Äî `high confidence`, `Tech U` education ‚Äî `low confidence`) each with a checkbox. Clicked Confirm ‚Üí redirected to `/profile` ‚Üí **the imported section actually appears in the section list alongside the manually-added one** ‚Äî the full HITL confirm round trip, live.
9. **Staleness badge, the other literal acceptance criterion**: added a document version carrying the profile's real current `factsHash` (via `profileFactsHash` on `POST .../versions` ‚Äî the one field the manual Add-Version form intentionally doesn't expose, since it's meant to come from an automated "generate from profile" flow, not a hand-typed value) ‚Üí documents list showed **"up to date"**. Then edited the profile (added a summary section, changing its `factsHash`) ‚Üí reloaded `/documents` ‚Üí the SAME document now showed **"stale"**. Verified live, not asserted from code reading.

One interaction-tooling quirk worth recording for future sessions in this environment: the `computer` tool's `left_click` on a `<button type="submit">` inside a React-controlled form did not reliably fire the click (no network request resulted, repeatedly, across multiple pages) ‚Äî while `form_input` for filling fields worked every time, and dispatching the click via `document.querySelector(...).click()` through the JS-execution tool worked every time. All form submissions above were driven that way; plain link navigation (`<a href>`) worked fine with `computer left_click`. This looks like a `computer` timing/targeting issue specific to this tool, not an application bug ‚Äî direct API calls (`curl`) and DOM inspection both confirmed the app's own code was correct throughout.
