# 030 — Class B: capture ingest endpoint

**Status:** DONE | **Milestone:** M4

## Objective
FR-3a: the LinkedIn/Indeed on-ramp. A user, already logged into their own account and viewing a job they're authorized to see, triggers "Capture to CareerPilot" (bookmarklet or extension — the extension/bookmarklet client itself is out of scope for this task; this task is the ingest endpoint it posts to). No stored credentials, no server-side automation.

## Files affected
- `apps/api/src/routes/capture.ts` (`POST /api/capture` — accepts an already-rendered page payload: URL + extracted title/company/description HTML/text)
- `packages/connectors/class-b/capture-ingest/index.ts` (`ConnectorPort` implementation: validates + normalizes the posted payload — this *is* the Class B "connector", it just receives pushed data instead of polling)
- `packages/contracts/src/capture.ts` (zod schema for the capture payload)

## Dependencies
026, 011

## Acceptance criteria
- [ ] Endpoint stores zero platform credentials — payload is exactly what the user's already-authenticated browser rendered, nothing is fetched server-side
- [ ] Rate-limited per-user (single-item, user-driven — reject bulk/scripted abuse per ADR-004's "one-job-at-a-time" posture)
- [ ] Normalizes into the same canonical `RawJob` shape as Class A connectors, feeding the same dedup path (029)
- [ ] Malformed/incomplete payload rejected with a typed 400, not a 500

## Test plan
Integration: POST a fixture payload (simulating what a real bookmarklet would extract from a rendered LinkedIn/Indeed page), assert it lands in `job_postings` with `source_connector_key = 'capture'` and dedups correctly against an equivalent Class A job.

## Test plan actually executed

Route mounted at **`POST /capture`**, not `/api/capture` as the task title literally says — every other route in this API (`/jobs`, `/auth/register`, `/applications`, `/board`) is unprefixed, so `/capture` matches the codebase's own established convention rather than the task file's exact string. Noted as a deliberate deviation, not an oversight.

Ran against real Postgres + Redis (`apps/api/test/integration/capture.test.ts`, 7 tests, all passing):
1. Requires auth (401 without a session).
2. A valid capture payload lands in `job_postings` with `source_connector_key='capture'`, correct title/company/description — and the test explicitly documents why zero credentials are involved: the payload IS the entire input, there is no server-side fetch step anywhere in the code path (`normalizeCapturePayload` takes the posted payload and returns a `RawJob`; nothing in `apps/api/src/routes/capture.ts` ever calls out to LinkedIn/Indeed/anywhere else).
3. A schema-malformed payload (missing url/description) → typed 400 (`application/problem+json`, `code: validation_failed`), not a 500.
4. A schema-VALID payload that still has no usable description (`CapturePayloadSchema`'s `.refine` catches most of this, but a payload with only whitespace description would pass zod and fail at `normalizeCapturePayload` instead) → still a typed 400, not a 500 — two independent validation layers, both proven.
5. **Cross-source dedup, the acceptance criterion's literal test**: seed a Class A (Greenhouse) posting for "Senior Backend Engineer" @ Acme, then capture a reformatted LinkedIn-shaped payload for the same job — both rows land with the same `dedup_group_id`, response status is `"duplicate"`.
6. Re-capturing the exact same URL twice is idempotent (`externalId` = SHA-256 of the canonicalized URL, same unique-index mechanism as every Class A connector) — second POST returns `"already_captured"`, row count stays at 1.
7. **Real rate-limit test, not asserted-by-reading-code**: 31 consecutive POSTs from the same authenticated user (varying URLs, so dedup isn't why later ones would be rejected) — the 31st returns HTTP 429. `keyGenerator` is per-user (`request.actor.userId`), not per-IP, matching ADR-004's "one person's pace of manual captures" framing.

Unit tests (`packages/connectors/src/class-b/capture-ingest/index.test.ts`, 8 tests): contract test-kit conformance (compliance class B), field normalization, idempotent externalId across two captures of the same URL, descriptionText preferred over descriptionHtml when both present, typed-error rejection of missing description / empty title.

**Full regression**: `turbo run typecheck` 10/10, `eslint .` 0 errors, full unit suite **237/237** (55 new: capture-ingest's 8 + everything already covered by 026-029), full integration suite run twice — capture.test.ts's 7 tests passed both times with zero flakiness; 1-2 UNRELATED pre-existing M2 concurrency tests (`outbox-relay.test.ts`'s 3-relay race, `end-to-end.test.ts`'s task-017 simultaneous-redelivery test) intermittently failed across the two runs (different tests failed each time, both passed cleanly when run in isolation) — confirmed via `git status` that neither underlying implementation file has any diff in this branch; this is this Docker container's CPU/IO throttling interacting with tight concurrency-timing assumptions in M2-era tests, reproduced identically with or without any M4 code present. Flagged, not silently ignored, not fixed (out of scope for M4).
