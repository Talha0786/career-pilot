# 041 — Diff-review UI (web)

**Status:** TODO | **Milestone:** M5

## Objective
Implement the M5 roadmap acceptance criterion "diff-review UI functional" and the docs/06-agent-design.md §4 point 4 requirement: "Human review is non-skippable before export/use. The UI shows a semantic diff against the base document and the claim→fact mapping." This is the user-facing gate that makes 040's `needs_human` flag meaningful rather than a dead-end state.

## Files affected
- `apps/web/src/app/` — new route for reviewing a generated `DocumentVersion` (follow the existing routing/page conventions from task 025's profile editor / document history UI — check `apps/web/src/app` structure before inventing a new pattern).
- `apps/web/src/components/` — a semantic diff view (base document content vs. generated version — reuse `@careerpilot/ui` components from task 018 where they fit; a plain structural diff over the `ResumeDocumentContent`/`CoverLetterDocumentContent` JSON, rendered as added/changed/unchanged per bullet, is sufficient — do not pull in a heavyweight text-diff library for a two-line change if the repo doesn't already have one).
- `apps/web/src/components/` — claim→fact mapping panel: for each flagged/unsupported claim from 040's `ClaimAudit`, show the claim text and either its mapped fact or an "UNSUPPORTED — needs your review" state, with an accept/reject action per claim.
- `apps/api/src/routes/documents.ts` (extend) — endpoints the UI needs: fetch a version's `ClaimAudit` + diff data, submit human review decisions (accept-as-is / edit / reject-and-regenerate), which then clears `needs_human` and unblocks export (per 040's export-blocking gate).
- `packages/contracts/src/documents.ts` (extend) — DTOs for the review payload.
- Tests: component tests if the repo has a component-testing setup already (check `apps/web` for existing test infra from task 025 before adding a new one), API integration test for the review-submission → `needs_human` cleared → export unblocked flow.

## Dependencies
- 039, 040 (needs real tailored documents with claim audits to review).

## Acceptance criteria
- A `needs_human: true` document version cannot be exported from the UI (button disabled/hidden with a clear explanation) until review is submitted.
- Reviewer sees, per flagged claim: the claim text, the semantic diff context it appears in, and either its fact mapping or its UNSUPPORTED status.
- Submitting a review (accept/reject per claim) updates the document version's review state and — for accepted-as-clean reviews — unblocks export.

## Test plan
- API integration: full round-trip — tailor (039) → verify (040, force an UNSUPPORTED fixture) → fetch review payload → submit review → assert export is now permitted.
- Manual browser verification (per this project's standing rule for UI changes): start the dev stack, walk the golden path (generate → review → export) and the edge case (reject a claim, confirm export stays blocked) in an actual browser before marking DONE.
