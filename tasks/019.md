# 019 — Domain: CareerProfile, ProfileSection, Document, DocumentVersion

**Status:** DONE | **Milestone:** M3

## Objective
Domain aggregates for the two M3 entities, following the same pattern as the existing `User`/`JobPosting`/`Application` aggregates (task 003): pure TS, no framework/infra imports, invariants enforced in constructors/methods, `Result`-returning where an operation can fail.

## Files affected
- `packages/domain/src/profile/career-profile.ts` (aggregate: sections list, embedding-stale flag)
- `packages/domain/src/profile/profile-section.ts` (entity: kind enum, sort, versioned JSONB content per kind — mirror design §2 `profile_sections`)
- `packages/domain/src/documents/document.ts` (aggregate: kind enum resume/cover_letter/other, currentVersionId, soft-delete)
- `packages/domain/src/documents/document-version.ts` (entity: append-only, version int, source enum imported/generated/edited, `profileFactsHash` for staleness detection per design §2 invariant)
- unit tests colocated per existing convention (`*.test.ts`)

## Dependencies
002, 003

## Acceptance criteria
- [x] `CareerProfile` enforces: sections belong to exactly one profile; section `kind` is a closed enum
- [x] `Document`/`DocumentVersion` enforce the append-only invariant at the domain layer (no `updateVersion` method — only `addVersion`)
- [x] `profileFactsHash` computed deterministically from profile content (used later by UI to flag stale documents — design §2)
- [x] Zero imports from `application`/`infrastructure` (boundary-check clean)
- [x] `tsc --noEmit` and `eslint` clean

## Test plan
Unit only (pure domain, no I/O): construction invariants, append-only enforcement, hash determinism.

## Test plan actually executed
Implemented `packages/domain/src/profile/{profile-section,career-profile}.ts` and
`packages/domain/src/documents/{document,document-version,document-content}.ts`,
plus new branded ids (`CareerProfileId`, `ProfileSectionId`, `DocumentId`,

`application.ts` (task 003).

Key design notes:
- `ProfileSection` is only ever constructed via `CareerProfile.addSection`,
  which stamps `this.id` as the `profileId` — there is no code path by which
  a section can be attached to more than one profile.
- `CareerProfile.isEmbeddingStale` is a **derived** getter (`embeddingStatus
  !== 'ready' || embeddedFactsHash !== factsHash`), not a redundantly stored
  boolean, so it can never drift from the facts it describes.
- `computeProfileFactsHash` is a pure function (sorted by section id, so it's
  independent of insertion/array order; `sort` is deliberately excluded from
  the hashed shape since reordering isn't a factual change) — reused by
  `DocumentVersion.profileFactsHash` at generation time for the staleness
  comparison used by task 025's UI badge.
- `Document`/`DocumentVersion` append-only: `Document` has no `updateVersion`
  method at all — `addVersion` always assigns `nextVersionNumber` (max
  existing + 1) and there is no parameter to target an existing version
  number, so "mutate an existing version" is structurally unreachable, not
  just discouraged. `DocumentVersion.withRenderedPdfKey` returns a *new*
  immutable instance (used by `Document.attachRenderedArtifact`) rather than
  mutating in place — the one field legitimately filled in after the fact by
  the task 024 renderer.
- Soft delete (`Document.softDelete`/`restore`) per the design rule "soft
  delete only where product requires undo (documents)".
