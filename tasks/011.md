## Deliberate deviation from the original test-plan wording
Cross-owner reads return **404 not_found**, not 403 forbidden. `JobPostingRepository.findByIdForUser` is documented as "no unscoped findById outside the worker path" — using `findByIdAnyOwner` from a route to produce a 403 would leak that the resource exists to a non-owner. 404 is the stricter interpretation and is what's actually implemented end to end (create-application, update-stage, get-job all agree). Documented here rather than silently diverging from the M2 design doc's one-line test-plan mention.

## Bug found and fixed during this task
`apps/api/src/routes/{jobs,applications}.ts` originally used `app.addHook('preHandler', requireAuth)` inside the route-registration function. Because the `app` instance is passed by reference (not through `fastify.register()` plugin encapsulation), this hook applied to the **entire Fastify instance**, not just those two route groups — `/healthz`, `/readyz`, and even the 404 handler started requiring auth. Caught by the integration tests (`/healthz` returned 401). Fixed by moving `preHandler: requireAuth` onto each route's options instead of a shared `addHook` call.
