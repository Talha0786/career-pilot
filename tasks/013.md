# 013 — Docker Compose stack

**Status:** DONE | **Milestone:** M2

## Objective
One command from clean clone to a running application. If this is hard, the project is dead on arrival for self-hosters.

## Files affected
- `docker-compose.yml`, `compose.dev.yml`, `Dockerfile` per app (`apps/api`, `apps/worker`, `apps/web`), `.dockerignore`, `Caddyfile`, `.env.example`, `packages/infrastructure/src/db/migrate.ts`

## Dependencies
011, 012

## Acceptance criteria
- [x] `cp .env.example .env && docker compose up` → healthy stack, **zero** manual steps
- [x] One-shot `migrate` service runs before `api` (`depends_on: service_completed_successfully`)
- [x] Healthchecks gate startup ordering; no race on first boot
- [x] Non-root users; multi-stage builds; pinned base image digests
- [x] `compose.dev.yml` runs Postgres + Redis only (apps via `pnpm dev`)
- [x] Boots with **no LLM key** (local Ollama default, ADR-006)

## What was built
- `docker-compose.yml` — 7 services: `postgres` (pgvector/pgvector:pg16), `redis`, one-shot `migrate`
  (runs `migrate.ts` against the `postgres` service, exits 0, gates `api`/`worker` via
  `service_completed_successfully`), `api`, `worker`, `web`, and `caddy` as the single-origin reverse proxy
  on port 80.
- `apps/{api,worker,web}/Dockerfile` — multi-stage (`base` → `deps` → `build` → `runtime`), all pinned to
  `node:24-alpine@sha256:a0b9bf06e4e6193cf7a0f58816cc935ff8c2a908f81e6f1a95432d679c54fbfd`, non-root
  `careerpilot` user, `apps/web` uses Next's `output: 'standalone'`.
- `Caddyfile` — `/api/*` and `/ws` routed to `api:8080`, everything else to `web:3000`, all from port 80 so
  the session cookie's `SameSite=Lax` never crosses an origin (M2 design §7).
- `compose.dev.yml` — Postgres + Redis only, for the `pnpm dev` inner loop.
- `packages/infrastructure/src/db/migrate.ts` — idempotent migration runner (tracks applied migrations in a
  `schema_migrations` table), reused by both the compose `migrate` service and `pnpm db:migrate`.

## Bugs found and fixed during live verification
All four of these were caught by actually running `docker compose up`, not by reading the compose file:

1. **`wget` healthcheck against `localhost` always failed with "connection refused"** (`api` and `web`
   services). Musl's `getaddrinfo` resolves `localhost` to `::1` first inside Alpine; the Node servers only
   bind IPv4 (`0.0.0.0`); BusyBox `wget` doesn't fall back to the IPv4 result on refusal. `api` sat
   "unhealthy" indefinitely (`FailingStreak` in the dozens) even though the process was up and answering
   fine on `127.0.0.1`. Fixed by pointing both healthchecks at `127.0.0.1` instead of `localhost`.
2. **Next.js standalone server bound to the wrong interface.** Docker auto-sets the `HOSTNAME` env var to
   the container ID, and `apps/web/.next/standalone/server.js` reads `process.env.HOSTNAME` as its bind
   address — so it was listening on `<container-id>:3000` instead of all interfaces, and `127.0.0.1:3000`
   inside its own container got "connection refused". Fixed by explicitly setting `HOSTNAME: 0.0.0.0` in
   the `web` service's environment in `docker-compose.yml`.
3. **Caddy forwarded `/api/*` verbatim instead of stripping the prefix.** The api service's own routes have
   no `/api` prefix (same convention `next.config.ts`'s rewrites already use), so `curl /api/healthz`
   through Caddy came back `404 not_found` even though `api:8080/healthz` worked directly. Fixed by adding
   `uri strip_prefix /api` to the Caddyfile's `/api/*` handle block. (Also had to `docker compose restart
   caddy` after editing — Caddy loads its config once at startup and doesn't hot-reload a bind-mounted file
   just because the file changed on disk.)
4. **`LLM_BASE_URL` from `.env` shadowed compose's Docker-appropriate default.** `.env.example` sets
   `LLM_BASE_URL=http://localhost:11434/v1` for the non-compose `pnpm dev` flow (same pattern as
   `DATABASE_URL`), but `docker-compose.yml` was interpolating that same variable
   (`${LLM_BASE_URL:-http://host.docker.internal:11434/v1}`) — since the var *was* set in `.env`, the
   fallback never applied, and `localhost` inside the `worker` container means the container itself, not a
   host-run Ollama. Fixed the same way `DATABASE_URL`/`POSTGRES_*` already split this: added a dedicated
   `DOCKER_LLM_BASE_URL` (empty by default, falls back to `http://host.docker.internal:11434/v1`) that
   compose reads instead.

## Test plan actually executed
- `docker compose down -v` (full teardown including the Postgres volume), `cp .env.example .env` +
  generated a fresh `SESSION_SECRET`, then a single `docker compose up -d` from a clean slate — all 7
  containers converged to healthy/exited-0 with no manual intervention, no `--build` needed on top of
  images already built once.
- Verified `migrate` ran exactly once, applied `0001_init.sql`, exited 0, and both `api`/`worker` waited on
  `service_completed_successfully` before starting.
- Verified through Caddy on port 80 (not a direct service port, to match how production is actually
  reached): `GET /api/healthz` → `{"status":"ok"}`, `GET /api/readyz` → postgres/redis both `true`,
  `GET /login` → 200 with real HTML.
- Confirmed `worker` boots and logs "worker + outbox relay running" with `LLM_API_KEY` empty and
  `LLM_BASE_URL` resolved to `http://host.docker.internal:11434/v1` — satisfies the "no LLM key required to
  boot" criterion (ADR-006). No live Ollama in this environment, so embed calls fail after boot, which is
  the expected/correct behavior, not a boot failure.
- Live browser smoke test through Caddy (port 80, single-origin, not a dev-mode proxy): registered a new
  account, redirected to `/board` with the session cookie set, pasted a job — it appeared instantly on the
  board via the outbox → BullMQ → worker → Redis pub/sub → WebSocket push chain, landing in "Discovered"
  with an "Embedding failed" badge (correct, since no Ollama is reachable).
- Re-ran the full suite against the compose-managed Postgres/Redis after the fixes: **162/162 unit tests**
  and **37/37 integration tests** (1 documented todo) passed. Integration tests need a separate
  `careerpilot_test` database and Redis DB index 2, per `SETUP_AND_USAGE.md` — this is a pre-existing local
  dev requirement independent of the compose stack (the compose Postgres only provisions the primary
  `careerpilot` database); created it manually against the running `postgres` container for this run.

## Known gaps / deviations
- No CI wiring yet to run this automatically on push — that's task 014.
- The Caddy "doesn't hot-reload on file change" behavior above is worth remembering if `Caddyfile` is
  edited again post-deploy: needs an explicit `docker compose restart caddy` (or `caddy reload` inside the
  container), not just a config edit.
- `host.docker.internal` was only live-verified on Windows/Docker Desktop (works natively there). Added
  `extra_hosts: - "host.docker.internal:host-gateway"` on the `worker` service so it also resolves on
  native Linux Docker, but that specific path is untested on this machine.
