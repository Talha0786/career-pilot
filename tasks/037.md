# 037 — Numbered fact-list compilation from CareerProfile

**Status:** TODO | **Milestone:** M5

## Objective
Implement the fact-base compiler at the heart of the anti-hallucination contract (`docs/06-agent-design.md` §4, point 1): "profile sections are compiled to a numbered fact list (`F1: "Led migration of X at Company Y, 2021–2023"`)." This is a pure function the tailoring pipeline (039) and claim verification (040) both depend on — build and unit-test it standalone first.

## Files affected
- `packages/domain/src/profile/fact-list.ts` (new) — `compileFactList(profile: CareerProfile): Fact[]`, `Fact { id: string /* "F1" */, sectionId: string, text: string }`. Granularity decision: compile at **bullet level, not section level** — a section like an `ExperienceContent` with 4 bullets should produce 4 facts (`F1..F4`), each independently verifiable, not one giant fact blob per job. This is the granularity claim verification (040) needs to point at specific unsupported claims. Reuse `ProfileSection.toContentText()` where it already flattens content, but split further where the section kind has an array of bullets/items.
- `packages/domain/src/profile/fact-list.ts` — deterministic ordering: **must** derive from the same canonical sort `computeProfileFactsHash` already uses (career-profile.ts:63), so the fact list and the hash stay in lockstep — a profile with an unchanged `factsHash` must always compile to the byte-identical fact list (this is what lets `DocumentVersion.profileFactsHash` mean "these exact facts were used").
- `packages/contracts/src/profile.ts` (or new `packages/contracts/src/facts.ts`) — `FactSchema = z.object({ id: z.string(), sectionId: z.string(), text: z.string() })`.
- `packages/domain/test/unit/fact-list.test.ts` (new).

## Dependencies
None beyond already-DONE task 019 (`CareerProfile`, `ProfileSection`, `computeProfileFactsHash`).

## Acceptance criteria
- `compileFactList` is pure (no I/O, no LLM call) and deterministic — same profile in, same fact list out, byte-for-byte, every call.
- Fact IDs are stable across calls for an unchanged profile (don't regenerate random IDs — derive `F{n}` from stable iteration order).
- Every `ProfileSection` kind (experience, education, project, skill_group, certification, summary) produces at least one fact; kinds with bullet arrays produce one fact per bullet.

## Test plan
- Unit: one fixture profile covering all section kinds, assert exact fact list output (golden-file style, like task 024's rendering tests — but plain data equality, not byte-file comparison).
- Unit: determinism — call twice on the same profile, assert identical output; call after a content mutation, assert IDs for unchanged bullets stay the same (only new/changed content gets new/changed facts) if the compiler design supports stable partial IDs — if it doesn't (e.g. if it's array-index-based and a reorder shifts every ID), document that as a known limitation in the task's Status note rather than silently shipping surprising behavior.
