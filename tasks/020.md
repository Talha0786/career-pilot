# 020 — Contracts + application ports/use-cases: profile & document CRUD

**Status:** DONE | **Milestone:** M3

## Objective
Zod DTOs and application-layer use cases for profile and document CRUD, mirroring task 004/005's pattern for `JobPosting`/`Application`.

## Files affected
- `packages/contracts/src/profile.ts`, `packages/contracts/src/documents.ts` (zod schemas: create/update profile, add section, create document, add version)
- `packages/application/src/ports/repositories.ts` (add `ProfileRepository`, `DocumentRepository` interfaces)
- `packages/application/src/profile/commands/{create-profile,update-profile,add-section}.ts`
- `packages/application/src/documents/commands/{create-document,add-document-version}.ts`
- `packages/application/src/documents/queries/{get-document,list-documents}.ts`

## Dependencies
019, 004, 005

## Acceptance criteria
- [x] Contracts package has zero framework deps (zod only), matching existing convention
- [x] Use cases take a `deps` object of ports (no direct infra import) — same DI shape as `embed-job-posting.ts`
- [x] `addDocumentVersion` rejects mutating an existing version (calls domain's append-only guard, surfaces as a typed error)
- [x] Unit tests use in-memory fake repositories, not real Postgres

## Test plan
Unit: one test per use case (happy path + at least one invariant-violation path), following the existing `packages/application/test/unit/*.test.ts` structure.

## Test plan actually executed

**Contracts** (`packages/contracts/src/profile.ts`, `documents.ts`): zod-only DTOs.
`AddSectionRequestSchema` and `DocumentContentSchema` are `z.discriminatedUnion('kind', ...)`
so a content shape that doesn't match its declared `kind` is rejected at the
edge, not downstream. `packages/contracts/src/profile-documents.test.ts` (7
tests) covers the discriminated-union mismatch case explicitly for both.

**Ports** (`packages/application/src/ports/repositories.ts`): added
`ProfileRepository`, `DocumentRepository`, extended `TransactionContext` with
`profiles`, `documents`, and a new `AuditPort` (`ports/audit.port.ts`,
`TransactionContext.audit`) — needed so task 022's "audit_log entry written
on document version creation" acceptance criterion can be satisfied
atomically in the same transaction as the version + outbox event, the same
pattern ADR-007 already established. `ProfileRepository` treats a career
profile as a **per-user singleton** in the API surface
(`findActiveForUser`, no profile id in `GET/PUT /api/profile` — task 022) —
documented in the port's docstring since it's a deliberate simplification
of the schema's `is_active`-per-row design.

**Use cases**: `profile/commands/{create-profile,update-profile,add-section}.ts`,
`profile/queries/get-profile.ts` (added beyond the literal file list —
needed by task 022's `GET /api/profile`, same shape as `get-job.ts`),
`documents/commands/{create-document,add-document-version}.ts`,
`documents/queries/{get-document,list-documents}.ts` (`get-document.ts` also
exports `makeGetDocumentVersionUseCase`, backing
`GET /api/documents/:id/versions/:versionId`).

Re: "addDocumentVersion rejects mutating an existing version" — there is no
`targetVersionId`/mutate parameter on `AddDocumentVersionInput` at all, so
"update version N" is structurally unreachable through this use case (same
reasoning as the domain layer, task 019). The concrete invariant-violation
path exercised is `Document.addVersion`'s own append-only guard surfacing
through unchanged: adding a version to a soft-deleted document returns the
domain's `conflict` error verbatim (`documents.test.ts`, "the domain
append-only guard rejects a version on a soft-deleted document").

New fakes added to `packages/application/test/fake-repos.ts`:
`FakeProfileRepository`, `FakeDocumentRepository`, `FakeAuditPort` — wired
into `FakeUnitOfWork`'s `TransactionContext`.

Verified in an isolated `node:22-slim` Docker container:
- `vitest run packages/domain packages/application packages/contracts` →
  **233/233 tests pass** (up from 164 after task 019 — 21 new application
  tests across `profile.test.ts` and `documents.test.ts`, 7 new contracts
  tests, all pre-existing suites still green).
- `tsc --noEmit` clean for all three packages.
- `eslint` clean for all three packages' `src`/`test`.
- `scripts/verify-boundary-enforcement.mjs` still passes both checks.

Note: extending `TransactionContext` with `profiles`/`documents`/`audit`
means `packages/infrastructure`'s `DrizzleUnitOfWork` (M2, task 007) no
longer satisfies the interface until task 021 implements the three new
adapters — expected and addressed there, not a regression in what M2
shipped (M2's own tests don't touch the new fields).
